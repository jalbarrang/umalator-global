# PRD: Full 9-Runner Race Simulation

> **Status:** MVP Complete
> **Last updated:** 2026-03-16
> **Route:** `/race-sim`

---

## 1. Overview

A new standalone tool at `/race-sim` — separate from and independent of the existing vacuum compare. Where vacuum compare runs two isolated single-runner races to measure basinn deltas, the race sim runs a true 9-runner race in a single shared `Race` instance where runners interact through position keeping, dueling, spot struggle, blocking, and overtaking. The vacuum compare and all existing tools remain untouched.

### MVP Scope

| Decision        | Choice                                                                               |
| --------------- | ------------------------------------------------------------------------------------ |
| Runner count    | 9 default runners (no outfits)                                                       |
| Skills          | None — default runners have set stats only                                            |
| Samples         | ≤ 10 per run                                                                         |
| Data collection | Skipped for now                                                                      |
| Conditions      | Exact (real spatial checks, not RNG approximations)                                  |
| Default field   | Randomizer for generating a 9-runner field with weighted strategy distribution        |
| Pacemaker       | Simpler selection (existing `getPacer()` logic) until 1.5 anniversary update arrives |

---

## 2. Goals

- Run a single `Race` instance with 9 runners that interact in a shared simulation
- Position keeping affects runner speeds based on pacer distance
- Dueling triggers from real proximity on the final straight
- Spot struggle triggers from real strategy-group proximity
- Blocked side and overtake use real spatial checks
- Produce finish orders for each sample

### Non-Goals (for MVP)

- Full visualization (velocity paths, bird's eye view, etc. — design TBD post-MVP)
- Skills, outfits, or user-configurable runners (Phase 2)
- Dynamic skill conditions (`near_count`, `is_surrounded`, etc. — Phase 2)
- Per-tick data collection or charting
- Web Worker infrastructure
- Progressive sample batching
- Runner state snapshotting for parallel-safe updates (Phase 3+)
- Any changes to vacuum compare, skill chart, uma chart, or skill planner

> **Note:** A temporary UI button on `/race-sim` IS in scope to trigger runs and display finish orders. Default field is 9 runners with 800 flat stats generated by the field randomizer (`generateMobField()`).

## 3. Architecture

### 3.0 Design Decision: Extend, Don't Duplicate

The existing `Race` and `Runner` classes are extended in place — no new subclasses or parallel implementations. The classes were already designed for this: `Race` manages a `Map<number, Runner>` with 9-gate assignment, `runnersPerStrategy`, and `getPacer()`, and `SimulationSettings` already has `mode: 'compare' | 'normal'`.

**Guard strategy to prevent leaking into vacuum compare:**

1. **Settings-level guards** — Position keeping is gated by `positionKeepMode`. Vacuum compare sets `positionKeepMode: 0`, and `applyVirtualPositionKeep` exits immediately. No poskeep code runs.

2. **Mode-level guards** — Dueling and conditions branch on `mode`. The `else` branch must be byte-for-byte identical to the code that runs today:

   ```
   if mode === 'normal' → proximityDueling() / exact conditions
   else                 → artificialDueling() / approximate conditions  (existing code, untouched)
   ```

3. **Identity initialization** — All new fields are initialized to their no-op/identity values unconditionally (both modes):
   - `posKeepSpeedCoef = 1.0` (multiplier identity — `targetSpeed *= 1.0` is harmless)
   - `positionKeepState = PositionKeepState.None` (same as the current hardcoded value in `updateLastSpurtState`)
   - `positionKeepActivations = []` (never pushed to in compare mode since `applyVirtualPositionKeep` doesn't run)
   - `posKeepNextTimer` ticks harmlessly in `this.timers` but is never read in compare mode

**Principle:** If every new field starts at its identity value and is only mutated inside mode-gated code, compare mode gets the exact same math it gets today — even if a guard is accidentally missed.

### 3.1 Engine Changes

All changes live in `src/lib/sunday-tools/`.

#### SimulationSettings

Add `positionKeepMode` to `SimulationSettings` in `common/race.ts`:

```typescript
positionKeepMode: number; // 0 = off, 2 = virtual
```

#### Runner — New Position Keep Fields

Add to the `Runner` class in `common/runner.ts`:

```
positionKeepState    : IPositionKeepState
posKeepSpeedCoef     : number
posKeepNextTimer     : Timer
posKeepExitDistance   : number
posKeepExitPosition  : number
posKeepMinThreshold  : number
posKeepMaxThreshold  : number
posKeepEnd           : number
positionKeepActivations : Array<PositionKeepActivation>
```

These match the `RunnerLike` interface already defined in `poskeep/virtual-position-keep.ts`.

#### Runner.onPrepare()

Call `initializePositionKeep(this, () => this.createTimer())` during `onPrepare()`, after `initializeSpotStruggle()`.

#### Runner.onUpdate() — Position Keep Integration

Insert `applyVirtualPositionKeep(this)` into the update loop between `updateRushed()` / `updateDownhillMode()` and `updateTargetSpeed()`:

```
updateHills()
updatePhase()
updateRushed()
updateDownhillMode()
processSkillActivations()
processTargetedSkillActivations()
applyVirtualPositionKeep()          ← NEW
updateDueling()
updateSpotStruggle()
updateLastSpurtState()
updateTargetSpeed()                 ← multiply by posKeepSpeedCoef
applyForces()
applyLaneMovement()
```

#### updateTargetSpeed()

Multiply base target speed by `this.posKeepSpeedCoef` when not in last spurt and not out of HP.

#### Wire `positionKeepState` into RaceStateSlice (3 locations)

Three places currently hardcode `PositionKeepState.None` that need to read `runner.positionKeepState` in full race mode (while staying `None` for vacuum compare via identity initialization):

1. **`Runner.updateLastSpurtState()`** — the `RaceStateSlice` passed to `healthPolicy.getLastSpurtPair()`
2. **`GameHpPolicy.extractRunnerState()`** — the `RaceStateSlice` used for per-tick HP consumption (critical: PaceDown gets `0.6x` HP modifier via `getStatusModifier`)
3. **`GameHpPolicy.getLastSpurtPair()`** — the `lastleg` slice used for last spurt HP estimation

#### updateDueling() — Real Proximity (mode === 'normal')

When `this.race.settings.mode === 'normal'`, replace `artificialDueling()` with real checks:

- Final straight only (already gated)
- 2+ runners within `3.0m` distance and `0.25 * courseWidth` lane gap
- 2s proximity timer
- Both in top 50% of the field
- Speed gap `< 0.6 m/s`
- Cannot trigger below `15%` HP
- Ends below `5%` HP (already implemented)
- Speed boost: `(200 * guts)^0.708 * 0.0001` m/s (already implemented)
- Accel boost: `(160 * guts)^0.59 * 0.0001` m/s² (already implemented)

`artificialDueling()` stays for `mode === 'compare'`.

#### Lane Movement — Direct Spatial Queries (mode === 'normal')

In vacuum compare, `blocked_side` and `overtake` are faked via `ApproximateStartContinue` with RNG start/continuation rates (a hack to simulate multi-runner proximity without actual runners present).

In the full sim this is unnecessary — blocking and overtaking are **emergent from real runner positions**. Instead of routing through the condition system, `applyLaneMovement()` computes them directly:

```
if mode === 'normal':
  sideBlocked = hasSideBlockingRunner()   ← spatial query against race.runners
  overtake    = isOvertakingRunner()      ← spatial query against race.runners
else:
  sideBlocked = getConditionValue('blocked_side')   ← existing approximate RNG (untouched)
  overtake    = getConditionValue('overtake')        ← existing approximate RNG (untouched)
```

- **`hasSideBlockingRunner()`:** Any other runner within ~1 horse lane laterally and within ~3–5m longitudinally ahead
- **`isOvertakingRunner()`:** This runner is passing another runner (within a few meters, moving faster, adjacent lane)

No new condition variants needed. The approximate conditions stay registered in compare mode exactly as they are today.

### 3.2 Default Runner Factory + Randomizer

New file: `src/lib/sunday-tools/race-sim/mob-factory.ts`

Produces default `CreateRunner` configurations. The term "mob" in the codebase refers to the default runner preset — no outfit, random mob portrait, 800 flat stats. Every slot in the field is a `CreateRunner`; "mob" is just the starting default.

```typescript
type MobConfig = {
  strategy: IStrategy;
  stats: StatLine;
  mood: IMood;
  aptitudes: RunnerAptitudes;
};

function generateMobField(): CreateRunner[]; // randomized 9-runner field with defaults
function createMobRunners(configs: MobConfig[]): CreateRunner[]; // from explicit configs
```

**`generateMobField()` — Field Randomizer:**

Generates a 9-runner field with weighted strategy distribution:

| #   | Strategy     | Default Stats |
| --- | ------------ | ------------- |
| 1–2 | Front Runner | 800 flat      |
| 3–5 | Pace Chaser  | 800 flat      |
| 6–7 | Late Surger  | 800 flat      |
| 8–9 | End Closer   | 800 flat      |

All defaults: `mood = 0` (normal), `skills = []`, aptitudes `A` across the board, no outfit.

### 3.3 Race Sim Entry Point

New file: `src/lib/sunday-tools/race-sim/run-race-sim.ts`

```typescript
type RaceSimParams = {
  course: CourseData;
  parameters: RaceParameters;
  runners: CreateRunner[]; // 9 runners
  nsamples: number; // ≤ 10
  masterSeed: number;
};

type FinishEntry = {
  runnerId: number;
  name: string;
  strategy: IStrategy;
  finishPosition: number; // distance at end of race tick
};

type RaceSimResult = {
  finishOrders: FinishEntry[][]; // one per sample
};

function runRaceSim(params: RaceSimParams): RaceSimResult;
```

Flow per sample:

1. `race.prepareRound(masterSeed + i)`
2. `race.run()`
3. Read `race.finishedRunners` in order → produce `FinishEntry[]`

## 6. Known Approximations

These are accepted trade-offs for the MVP:

- **Sequential runner updates:** Runners update in Map insertion order within each tick. Runner N sees runners 0..N-1's updated positions for the current tick, but runners N+1..8's positions from the previous tick. At 15 FPS (~1m drift per tick at typical speeds) this is negligible. A snapshot-based approach for parallel-safe updates is deferred to Phase 3+.
- **Finish order tie-breaking:** If two runners cross the finish line on the same tick, the one earlier in iteration order is recorded first. The real game likely breaks ties by overshoot distance (`runner.position - course.distance`). Minor for MVP.

## 8. Future Phases (Out of Scope)

### Phase 2: Skills, Outfits & Dynamic Skill Conditions

- Allow users to assign outfits and skills to any runner
- Field randomizer includes outfit/skill presets
- **Two-tier condition lookup:** Create a `dynamicConditions` registry for conditions that have real runner-data implementations. At evaluation time, check `dynamicConditions` first; if not found, fall back to the existing static approximations (`noopErlangRandom`, `noopImmediate`, etc.). This makes the rollout incremental — implement high-value conditions first (`near_count`, `order_rate`, `is_surrounded`), and everything else still works via approximations until it gets a dynamic version. Conditions that may never have a meaningful dynamic implementation (`popularity`, `is_exist_chara_id`, etc.) stay on the static side permanently.
